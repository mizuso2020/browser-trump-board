### 実装の要点

- ルームAPIで参加者を同期（ポーリング 400ms、`Cache-Control: no-store`）
- 参加時に `player` パラメータ付きURLでプレイヤーIDを保持（localStorage / sessionStorage / Cookie）
- 役職など秘密情報は `hostSecrets` / `playerSecrets` に分割保存
- ホストはロビーで構成選択 → 「人狼を始める」で全員へ `wolf_ready`（役職確認）へ

### ついでに調整したUX

- 議論時間 **5分 → 3分**
- ゲーム終了時の **役職一覧を参加者全員が閲覧可能**（`revealedRoles` を公開データに保存）

## てこずったバグ3件（原因と対策まとめ）

今回いちばん時間がかかったのは **「ゲーム開始後、参加者1人だけロビーに残る」** 問題。  
加えて対策の副作用と、人狼役だけのクラッシュが重なった。

### 1. 一人だけロビーに残る（メインのバグ）

| | 内容 |
|--|------|
| **症状** | ホストが「人狼を始める」後、参加者のうち **ランダムに1人だけ** ロビー待機のまま。ブラウザ・参加順・待ち時間は無関係。強制再読み込みでも直らないことがある |
| **原因** | ホストがゲーム開始を保存した **後** に、遅れて届いた **古いロビーデータの PUT** がサーバーを上書きし、状態が `wolf_ready` → `lobby` に戻っていた |
| **なぜホストだけ進めるか** | ホストは開始ボタンで **ローカル画面を即切替**。参加者は **サーバーの状態をポーリング** して切替えるため、上書きのタイミングで取り残される |
| **対策** | `room_server.py` で **ゲーム開始後はロビーへ戻す保存を拒否**（`reject_public_put`）。クライアント側は同期の直列化・`gameStartedAt`・取り残し時の自動リカバリ（`hardRecoverToRoom`） |

### 2. 「保存が古くなりました」でルームが作れない（対策の副作用）

| | 内容 |
|--|------|
| **症状** | 「ルームを作る」直後に `保存が古くなりました。画面を更新してください` で落ちる |
| **原因** | 上記対策の「古い保存拒否」が **ロビー作成直後の正常な保存** までブロック。スマホとサーバーの **時刻ずれ** で `updatedAt` が古いと判定された |
| **対策** | **ロビー中は厳しいチェックをしない**。**ゲーム開始後だけ** 古い保存・フェーズ逆行を拒否。クライアントは `updatedAt` を `max(前回+1, Date.now())` で送信 |

### 3. 人狼を始めると一人だけエラー（`cgq58v3u` 等のプレイヤーID）

| | 内容 |
|--|------|
| **症状** | `Cannot read properties of undefined (reading 'プレイヤーID')` でエラー画面 |
| **原因** | オンラインでは全員の役職一覧（`roles`）は公開しない。**人狼役の参加者** だけ、仲間表示で存在しない `roles` オブジェクトを参照してクラッシュ |
| **なぜホストは大丈夫か** | ホストは `hostSecrets.roles` で全員分を持っている。参加者の人狼だけ `roles` が空 |
| **対策** | `getRoles` の安全化、人狼の仲間は `playerSecrets.wolfMates` から表示、`roleConfirmed` の未定義ガード |
